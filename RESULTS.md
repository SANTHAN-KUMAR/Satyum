# Testing, evaluation, and results

This document is deliberately separate from the pitch. It says what's actually been measured,
what the methodology is for measuring the rest, and where a number would need to be filled in
before anyone should repeat it to a judge. Nothing here is invented. Where a metric hasn't been
run yet, it says so.

## 1. What "passing a test" is required to mean here

Per `CLAUDE.md` section 3.2, a test in this codebase has to prove discrimination, not existence. A
test that would still pass if you replaced the analyzer with `return <constant>` is banned. Every
detector ships with a genuine-vs-tampered pair it has to tell apart, plus a set of must-fail
fixtures (a self-signed certificate must fail chain validation, an appended-bytes PDF must fail
signature coverage, a single edited balance figure must break arithmetic). This is enforced by a
constant-return guard that monkeypatches each detector to a fixed output and checks that its own
tests then fail. The full attack matrix and rationale live in
[`architecture/TESTING-STRATEGY.md`](architecture/TESTING-STRATEGY.md).

## 2. The automated test suite

The backend test suite currently defines **1,758 tests across 64 files** (`pytest --collect-only`
in `backend/`, counted directly from source, verifiable by anyone who clones the repo). That
includes:

- Unit tests per analyzer (signature verification, arithmetic, OCR cross-read, copy-move, perceptual
  hash, entity extraction, rule packs).
- Integration tests for the orchestrator, the risk engine, and the API routes.
- The must-fail fraud fixtures described above.
- The constant-return guard itself.
- An end-to-end pass over the curated real-document corpus (`test_e2e_real_corpus.py`).
- A much larger combinatorial pass (`test_e2e_mass_corpus.py`) that runs against a locally generated
  corpus of 1,000+ programmatically forged document variants, see section 4.

Run it with:
```bash
cd backend && source .venv/bin/activate
python -m pytest -q --ignore=tests/test_e2e_mass_corpus.py
```
or the full one-command version, which also regenerates the synthetic corpus first so nothing in it
is hand-edited:
```bash
./scripts/verify-satyum.sh
```

**Honest note on what's in this document versus what isn't:** at the time of writing, a full run of
the suite wasn't completed within a reasonable time budget in this session (some end-to-end tests
make live calls to the configured VLM provider and take minutes, not seconds). Rather than quote a
partial or stale pass count, this document reports the one thing that was independently verified: a
single real end-to-end call through the running API (section 3). Anyone can get a current, complete
pass/fail count for their own environment and API keys by running the command above.

## 3. One real, observed end-to-end result

To ground this document in something actually run rather than described, one real API call was made
against a live instance of the backend during this session:

```
POST /api/verify, file: demo/02_statement_tampering/tampered_salary_inflate.pdf
Result:   verdict = REJECTED, trust_score = 58.5
Signal:   financial_consistency = VALID (fired, and correctly identified the reconciliation break)
Latency:  20.7 seconds, wall clock, including a live VLM call
```

That single data point demonstrates the pipeline working end to end (a tampered document submitted
over HTTP, read, judged, and rejected with a named signal) and gives one honest latency figure:
around 20 seconds for a document that requires a live vision-language model read. A document that
verifies purely on its digital signature (no VLM call needed) is expected to be much faster, on the
order of the PDF parse plus a chain-validation check, but that specific number wasn't separately
timed in this session and shouldn't be quoted until it is.

## 4. The synthetic and generated test corpora

Three separate corpora exist, at different scales and for different purposes:

- **`samples/`** (git-tracked, small): the hand-picked set used in `samples/README.md`, signed and
  unsigned PDFs, a consistent identity bundle, a mismatched identity bundle. Fast, always present,
  used in CI-style runs.
- **`samples/real_corpus/`** (git-tracked): statements and identity documents derived from real
  Canara Bank statement layouts and Aadhaar formats (public templates, not a real person's data),
  each with a documented forgery type and expected verdict in
  [`samples/real_corpus/README.md`](samples/real_corpus/README.md). This is what `demo/` was curated
  from.
- **`samples/mass_corpus/`** (local only, not committed, regenerated by
  `samples/generate_mass_corpus.py`): a combinatorial corpus of 500 statement variants and 500
  identity-document variants, built by applying different forgery techniques (figure inflation,
  digit typos, date changes) on top of source templates, for 1,000+ total documents. This corpus is
  real and does exist locally at that scale. It is not part of the git repository for two reasons:
  its current generator is seeded from a real personal bank statement and Aadhaar PDF rather than a
  synthetic template (a fix worth making before this corpus is used again), and at 1,000+ PDFs it's
  too large to be a reasonable thing to ship in source control.

  The last recorded run of this corpus against the pipeline predates several extraction and
  arithmetic fixes made later in development, so its stored results are stale and are not repeated
  here. Re-running it is a legitimate way to get a large-N precision/recall number, but it makes
  1,000+ live calls to whichever document-reading provider is configured, which costs real API usage
  and time, so it should be a deliberate, budgeted decision rather than something run casually.

## 5. How to actually evaluate a system like this

A trust score and a verdict are not, by themselves, evidence that a system works. Here is the
methodology this project uses (and that a judge or an evaluator should ask for), broken into what's
already implemented as tests versus what a larger validation exercise would add.

### 5.1 Detection metrics, on a labeled corpus

Given a corpus where each document is labeled genuine or tampered (as all three corpora above are):

- **Recall on tampered documents** (also called the true positive rate): of the documents that were
  actually tampered, what fraction did the system flag as REJECTED (or route to REVIEW, which is
  also a correct non-approval)? This is the number that matters most for a fraud system, a forgery
  that sails through as APPROVED is the failure mode with the worst consequences.
- **False positive rate on genuine documents**: of the documents that were genuinely clean, what
  fraction got wrongly REJECTED? This is the number that matters for the business, every false
  rejection is friction for a legitimate customer.
- **Precision and F1**, if a single combined number is wanted, though in a banking context recall on
  fraud and the false-positive rate usually matter more separately than a blended score does, because
  the cost of a missed fraud and the cost of an annoyed customer aren't the same and shouldn't be
  averaged together.
- These should be reported **per signal** as well as overall (arithmetic consistency alone, identity
  corroboration alone, signature verification alone), because a system that is strong on one signal
  and silent on another looks very different from a system that is moderately good everywhere.

### 5.2 Test-suite quality, not just test-suite size

Line coverage is explicitly rejected as a metric in this project (`TESTING-STRATEGY.md` section 1),
because it's possible to execute every line of a detector and assert nothing meaningful. The metric
used instead is **mutation score**: take the detector's code, mutate it in small ways (flip a
comparison, swap an operator, drop a check), and see whether the existing tests catch the mutation.
A test suite with a high mutation score on the security-critical modules (signature verification,
arithmetic, risk aggregation) is doing real work; a suite that lets most mutants survive is not,
regardless of how many lines it touches.

### 5.3 Fail-closed rate under injected failure

Chaos-style testing: deliberately break a dependency (a timeout, a malformed response from the
document reader, a corrupted upload) and confirm the system degrades to REVIEW or an explicit error,
never to a silent APPROVE. The relevant number is not "does it crash" but "does every failure path
resolve to the safe side."

### 5.4 Latency, by tier

Because the pipeline has genuinely different-cost tiers (a cryptographic signature check is
milliseconds; a vision-language model read of a multi-page document is seconds), latency should be
reported per tier reached, not as one blended average. A document that verifies on its signature
alone should be reported separately from one that needs a full VLM read plus cross-document
corroboration.

### 5.5 Audit and reproducibility checks

Two checks that are structural rather than statistical: does `/api/health` report
`audit_chain_intact: true` after a sequence of verdicts, and does editing any past audit record
break the chain at that point (proving the log is tamper-evident, not just append-only in name).

## 6. On comparing this to manual underwriting

It's tempting to put a number on how many times faster or more efficient this is than a human
underwriter reviewing the same document, and that number would make for a better slide. It isn't
included here, because no real, measured baseline for manual review time exists in this project, and
inventing one would be exactly the kind of fabricated metric this project's own engineering charter
forbids (`CLAUDE.md` section 3, "never invented numbers").

What can be said honestly, and is arguably the more interesting claim anyway: a human reviewer
skimming a multi-page bank statement is not going to recompute every running balance by hand, is not
going to cryptographically verify a PDF signature's certificate chain, and is not going to check a
document's perceptual hash against every other document the bank has ever seen. Satyum does all
three, deterministically, on every submission, every time. The advantage isn't best expressed as a
speed multiplier against a baseline nobody measured. It's that the checks it performs are checks a
manual review realistically never performs at all, at any speed, because they aren't practical to do
by hand at scale.

If a real manual-review time baseline exists (a Canara Bank process metric, a published industry
figure), it can be dropped into this section with a citation, and an honest comparison number
follows immediately from that.

## 7. What isn't measured yet, on record

Being explicit about gaps, rather than quietly omitting them:

- No full, current run of the 1,758-test backend suite is quoted in this document (section 2).
- No fresh precision/recall run against the 1,000+ document mass corpus is quoted (section 4);
  the last one is known to be stale.
- No per-tier latency breakdown exists yet, only the single 20.7-second end-to-end sample in
  section 3.
- No real manual-underwriting baseline exists to compare against (section 6).

Each of these is a well-defined, bounded piece of follow-up work, not an open-ended one, and each
should be run and the real number substituted in before this document is presented as a complete
results record rather than a methodology plus one verified data point.
